1. What is memoization

Memoization = remember the last answer; if asked the same question, return the saved answer instead of recomputing. 

// Without memoization: runs the heavy work EVERY call
function slow(n) { return heavyCompute(n); }

// With memoization: remembers last input + output
function memo(fn) {
  let lastArg, lastResult, has = false;
  return (arg) => {
    if (has && arg === lastArg) return lastResult; // HIT: skip work
    lastArg = arg; lastResult = fn(arg); has = true;       // MISS: compute + save
    return lastResult;
  };
}

Two payoffs, both critical here:

  • Speed — skip expensive recomputation.
  • Reference stability — return the same object in memory as last time. This is what stops React re-renders (explained in §8).

2. What is LRU

LRU = Least Recently Used. A cache that holds the last N entries. When full, it evicts the entry used longest ago. maxSize: 1 (the default here) means "remember only the most recent call". 

// From your new lruMemoize.ts — the LRU "move to front" on a hit:
if (areArgsEqual(entry.args, args)) {
  if (i > 0) {
    entries.splice(i, 1);   // remove from current spot
    entries.unshift(entry);  // put at front = "most recently used"
  }
  return entry.value;       // HIT
}
// ...after computing a MISS:
entries.unshift({ args, value });
if (entries.length > maxSize) entries.pop(); // evict least-recently-used

Front of the array = freshest. Back = stalest, evicted first. With maxSize: 1 there is exactly one slot, so any different argument is a miss.

3. The Store: where all data lives

Each Data Grid and each Chart owns one Store — a single object holding all state, plus a list of listeners. 

// Store.ts (simplified)
class Store {
  state;                  // the one big state object
  listeners = new Set();  // components waiting for changes

  setState(newState) {
    this.state = newState;
    this.listeners.forEach(l => l(newState)); // notify everyone
  }
  subscribe(fn) { this.listeners.add(fn); ... }
}

Grid store shape

// GridStateCommunity — one key per feature (~24 keys)
{
  rows:         { dataRowIds: [...], dataRowIdToModelLookup: {...} },
  columns:      { orderedFields: [...], lookup: {...} },
  filter:       { filterModel: {...}, filteredRowsLookup: {...} },
  sorting:      { sortModel: [...], sortedRows: [...] },
  pagination:   { paginationModel: { page, pageSize } },
  rowSelection: [...],
  focus:        { cell, columnHeader },
  dimensions, density, virtualization, ...
}

Chart store shape

// ChartState — merged from all active plugins
{
  series:        { defaultizedSeries, idToType, dataset },
  seriesConfig:  { config },
  dimensions:    { width, height, margin },
  animation:     { skip },
  // optional feature plugins add their own keys:
  interaction:   {...},   zoom: {...},
  brush:         { start, current },
  cartesianAxis: { xAxis, yAxis, xAxisExtrema },
}
The single most important rule. Every update creates a new top-level object but keeps the references of slices it did not touch: 
// update only filter → new state object, but state.sorting is the SAME ref as before
store.setState({ ...state, filter: newFilter });
This reference-preservation is the entire foundation memoization stands on.

4. The problem selectors solve

A component rarely needs the whole store — it needs a derived slice: "the rows after filter + sort", "the visible series". Computing that is often expensive and produces a new array/object each time.

If you recompute on every store change, two things break:

  • Wasted CPU re-deriving data that did not change.
  • A fresh []/{} every time is a new reference → React thinks it changed → re-renders needlessly.

A selector = a function (state, ...args) => derivedValue. A memoized selector solves both problems at once.

5. Two functions — and only one memoizes

  createSelector createSelectorMemoized
Caches? NO YES
Does Just composes: run inputs, run combiner, return. Composes + wraps in the two-layer cache.
Use for Cheap results / primitives (number, boolean). Expensive results / objects & arrays.
Why Recompute is free; primitives compare by value anyway. Avoid recompute + keep stable reference to stop re-renders. 
// createSelector — NO cache. Builds a plain function:
selector = (state, a1, a2, a3) => {
  const va = a(state, a1, a2, a3);   // run input selector a
  const vb = b(state, a1, a2, a3);   // run input selector b
  return c(va, vb, a1, a2, a3);      // run combiner — EVERY call
};
Read this snippet carefully — it is what createSelector returns. createSelector does not select anything itself; it assembles a function out of the pieces you passed and hands it back. Above is the assembled function for the "2 input selectors + 1 combiner" case (createSelector(a, b, c)):
  • a, b = input selectors — each pulls one slice out of state (e.g. state.filter, state.sorting).
  • c = combiner — turns those slices into the final derived value.
  • va, vb = the values the input selectors returned, fed into the combiner.
The // EVERY call comment is the whole point: nothing caches this — a, b, c all re-run on every invocation. (The real file has parallel branches for 1–8 selectors; this is the a && b && c one. createSelectorMemoized builds the same shape, then wraps it in the two cache layers.)

Real example — canItemBeReordered

From the Tree View Pro (x-tree-view-pro/.../itemsReordering/selectors.ts). Exactly the createSelector(a, b, c) shape, and it uses the non-memoized createSelector because the result is a boolean — cheap to recompute, no reference to stabilize.

  labelSelectors.isAnyItemBeingEdited,   // ← b: input selector (reused from another file)

Note the extra argument itemId: it is not produced by an input selector — it is passed in at call time (store.use(canItemBeReordered, itemId)) and forwarded straight to the combiner. That matches the a1, a2, a3 slots in the assembled-function snippet above.

Contrast — same file, memoized sibling: draggedItemProperties uses createSelectorMemoized because it returns an object ({ id, ... }). Object output must keep a stable reference across calls, or every subscriber re-renders — exactly the case the two cache layers exist for.
The two layers below only exist inside createSelectorMemoized. createSelector never touches lruMemoize or weakMapMemoize.

6. The two cache layers

createMemoizedSelector wires two memoizers together — exactly mirroring what reselect did: 

// createSelector.ts
function createMemoizedSelector(selectorsAndCombiner, options) {
  const combiner       = selectorsAndCombiner.at(-1);
  const inputSelectors = selectorsAndCombiner.slice(0, -1);

  const memoizedCombiner = lruMemoize(combiner, { equalityCheck: Object.is });  // INNER

  return weakMapMemoize((...args) => {                                     // OUTER
    const values = inputSelectors.map(s => s(...args));  // pull slices from state
    return memoizedCombiner(...values);
  });
}

Outer — weakMapMemoize (keyed on the arguments)

Builds a tree of caches, one level per argument. Object args (like state) go in a WeakMap; primitives in a Map. Walk the tree by args → if the leaf was computed, return it without running anything. 

// weakMapMemoize.ts — per-argument descent
let node = root;
for (const arg of args) {
  const map = isObjectLike(arg) ? node.objects : node.primitives;
  node = map.get(arg) ?? createCacheNode(); // follow / create branch
}
if (!node.computed) { node.value = func(...args); node.computed = true; }
return node.value;

Why WeakMap? When the state object is garbage-collected, its cache branch vanishes automatically — no memory leak, no manual cleanup.

Picture the cache as a tree. Each node holds two maps — objects (a WeakMap, for object args) and primitives (a Map, for things like numbers/strings) — plus a value slot at the leaf. Calling the selector means walking down one branch, one argument at a time:

// Calls made over time, each = selector(state, axisId): // selectXAxisTicks(stateA, "x") // selectXAxisTicks(stateA, "y") // selectXAxisTicks(stateB, "x") root (a CacheNode) │ .objects = WeakMap // next arg is an object ┌─────────┴──────────┐ key=stateA (obj) key=stateB (obj) ← arg #1: the state object │ node │ node │ .primitives = Map │ .primitives = Map // next arg is a primitive ┌────┴─────┐ │ key="x" key="y" key="x" ← arg #2: the axisId │ leaf │ leaf │ leaf ▼ ▼ ▼ computed=true computed=true computed=true ← value cached here value=[ticks] value=[ticks] value=[ticks]

How each call resolves against this tree:

Call Walk Leaf Result
(stateA, "x") again root → objects[stateA] → primitives["x"] already computed HIT — return value, run nothing
(stateA, "z") root → objects[stateA] → primitives["z"] missing new leaf, computed=false MISS — compute, store at new leaf
(stateC, "x") root → objects[stateC] missing whole new branch MISS — compute, store
The garbage-collection trick. Because the first level is a WeakMap, the stateA branch is only reachable through the stateA object itself. The moment stateA is no longer referenced anywhere, the GC reclaims it and its entire cache branch (both axis leaves) disappears automatically — no eviction logic, no size cap, no leak. This is why the outer layer uses a weak tree (unbounded but self-cleaning) while the inner lruMemoize uses a fixed maxSize.

Inner — lruMemoize (keyed on the input values)

Wraps the combiner. Compares the freshly-pulled input values against the last call with Object.is. Same values → return the old result object (stable reference). Different → recompute.

Why two layers, not one?
  • Outer (weakMap, on args): instant bail-out when called with the identical state again, and leak-free lifetime.
  • Inner (lru, on values): even when state is a brand-new object but the slice you read is unchanged, you still get back the same result reference.

8. useSyncExternalStore: the React bridge

The Store lives outside React — it is a plain class, not React state. useSyncExternalStore is the official React 18+ hook for letting a component read from and subscribe to an external store safely. It is the glue in steps 4–7 of the flow.

The hook's contract

const value = useSyncExternalStore(
  subscribe,         // 1. how to register a re-render callback; returns an unsubscribe fn
  getSnapshot,       // 2. how to read the current value (client)
  getServerSnapshot, // 3. how to read it during SSR (optional)
);

What React does with these three:

  1. On mount, calls subscribe(cb) — React's cb gets added to the store's listener set.
  2. To render, calls getSnapshot() to get the current value.
  3. When the store fires cb, React calls getSnapshot() again and compares the new return to the previous one with Object.is. Same reference → no re-render. Different → re-render.
This comparison is the exact moment memoization pays off. getSnapshot here is your selector running against the latest state. If the selector returns the same reference (a memoization HIT), Object.is is true and React skips the component — §9 is this rule applied to multiple components.

How MUI X wires it — useStore.ts

export function useStore(store, selector, a1, a2, a3) {
  return useStoreImplementation(store, selector, a1, a2, a3);
}

// React 19+: the raw hook
function useStoreR19(store, selector, a1, a2, a3) {
  const getSelection = React.useCallback(
    () => selector(store.getSnapshot(), a1, a2, a3), // getSnapshot = run selector on state
    [store, selector, a1, a2, a3],
  );
  return useSyncExternalStore(store.subscribe, getSelection, getSelection);
}

// React 17/18: the "with selector" variant from the shim
function useStoreLegacy(store, selector, a1, a2, a3) {
  return useSyncExternalStoreWithSelector(
    store.subscribe,
    store.getSnapshot,          // raw state snapshot
    store.getSnapshot,          // SSR snapshot
    (state) => selector(state, a1, a2, a3), // selector applied separately
  );
}

Two implementations, why

  useStoreR19 (React ≥19) useStoreLegacy (React 17–18)
Hook useSyncExternalStore useSyncExternalStoreWithSelector (shim)
Selector applied inside getSnapshot itself as a separate selector argument
Why raw hook is reliable here on R19+ raw hook had edge-case issues on R18 in tests; the shim + React 17 fallback are needed for older support

The choice is made once: reactMajor >= 19 ? useStoreR19 : useStoreLegacy.

Why this hook and not useState/Context

  • External source of truth — the store outlives renders and is mutated imperatively by plugins/handlers; it is not React state.
  • Tearing-safe — in concurrent React, two components could otherwise read different store values mid-render. useSyncExternalStore guarantees a consistent snapshot, preventing this "tearing".
  • Granular subscriptions — each component subscribes through its own selector, so each re-renders only when its slice changes. Context would re-render every consumer on any change.

9. How only PART re-runs, not the whole component

This is the payoff. React's useSyncExternalStore works like this: after the store notifies, it calls your selector and compares the returned value to the previous one with Object.is. If they are the same reference, React treats the hook output as unchanged and does not re-render that component. 

// A component subscribes to ONE slice:
function RowList() {
  const rows = store.use(gridFilteredSortedRowEntriesSelector);
  return rows.map(...);
}
function DensityButton() {
  const density = store.use(gridDensitySelector);
  ...
}

One setState notifies both components. But:

  • Change state.filter → RowList's selector returns a new array (re-renders); DensityButton's selector returns the same density (skips).
  • Change state.density → reverse: DensityButton re-renders, RowList skips.

"Only part re-runs" has two distinct meanings — both happen:
  1. The computation: memoization skips re-deriving slices whose inputs are unchanged (§6).
  2. The render: because skipped computations return the same reference, the subscribing component's hook sees no change and React skips that component's render (this section).
Memoization is the mechanism; selective re-rendering is the result. No memoization → new reference every time → every subscriber re-renders on every change.

11. One-paragraph recap

Each grid/chart keeps all state in a Store. On any interaction, setState builds a new top-level object but preserves references of untouched slices. Components subscribe to derived slices through selectors. createSelector just composes (no cache — for cheap primitives); createSelectorMemoized adds a two-layer cache: weakMapMemoize (outer, keyed on arguments, leak-free) for instant bail-out, and lruMemoize (inner, keyed on input values) so unchanged slices return the same result reference. React's useSyncExternalStore compares that reference and re-renders only the components whose slice actually changed. This branch replaces the external reselect library with in-house copies of those two memoizers — same behavior, one fewer dependency.
