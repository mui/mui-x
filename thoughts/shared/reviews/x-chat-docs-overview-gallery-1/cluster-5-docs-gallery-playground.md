## Findings

### 1. [MEDIUM] Classes-editor textarea has no accessible label / no error wiring

`docs/src/modules/components/chat-playground/PlaygroundCard.tsx:693-743`

The Classes editor is a raw `<Box component="textarea">`. Its "label" is a separate clickable
`<Typography>` disclosure header (`PlaygroundCard.tsx:660`), not a `<label htmlFor>`, and the
textarea has **no `aria-label`**. When a parse error occurs it is shown only as a sibling
`<Typography color="error">` (`:730-743`) with red border styling (`:717,725`) — there is no
`aria-invalid`, and the error text is not connected via `aria-describedby`.

Why it matters: screen-reader users get an unlabeled multi-line text field, and the validation
error (the whole point of `parseSx` surfacing `parseError`) is never announced. This is a public
docs page.

Fix: give the textarea `aria-label={item.selector ?? item.name}`, add
`aria-invalid={Boolean(item.parseError)}`, give the error `<Typography>` an `id` and reference it
with `aria-describedby` when present.

### 2. [LOW] Generated `generatedCodeSources.ts` is not reproducible from its own generator

`docs/scripts/generateChatOverviewCodeSources.mjs` → `docs/src/modules/components/overview/chat/mainDemo/generatedCodeSources.ts`

Re-running `node docs/scripts/generateChatOverviewCodeSources.mjs` produces a diff vs. the committed
file. The **string contents are identical** — the only difference is line wrapping: the generator
emits `export const x = "..."` on one line, but the committed file was post-processed by Prettier
into `export const x =\n  "..."` (commit "Format chat overview playground demos with Prettier").

Why it matters: the file header says "Generated by ... Do not edit this file directly," yet the
committed form is _not_ what the generator emits, so any contributor who regenerates + commits will
introduce churn, and a CI "check generated files are up to date" guard (if/when added) would fail
unless it also runs Prettier. The script should pipe its output through Prettier (or the contract
should document "run the generator then `pnpm prettier`"). Not a content bug.

### 3. [LOW] Playground demos import from `docs/src/...`, so the CodeSandbox/StackBlitz export is broken

e.g. `docs/data/chat/basics/chatbox/ChatBoxPlayground.tsx:13`
(`import { PlaygroundCard } from 'docs/src/modules/components/chat-playground/PlaygroundCard'`),
52 demo files total import from `docs/src/modules/components/chat-playground/*`.

These demos are embedded with `{{"demo": "...Playground.js", "bg": "inline", "defaultCodeOpen": false}}`
but do **not** set `disableCodeSandbox` / `disableStackBlitz`. The "Open in CodeSandbox/StackBlitz"
buttons therefore remain, and a standalone sandbox cannot resolve `docs/src/...` paths → broken
export.

Mitigating context (why this is LOW, not HIGH): there is established precedent in the same repo —
`docs/data/date-pickers/playground/PickersPlaygroundWrapper.js` and several
`docs/data/charts/legend/*` demos import from `docs/src/modules` the same way. So this is an
accepted MUI X docs pattern for "in-docs-only" playground wrappers; the in-page render works fine.
If a clean sandbox export is desired, set `disableCodeSandbox`/`disableStackBlitz` on these demos.

### 4. [NIT] `parseSx` silently drops invalid blocks from the live preview

`useCustomizations.ts:93-101` (`toClassesSx`) skips any block whose `parsed.value` is falsy, and
`:72-80` surfaces `parsed.error` per field.

So on a parse error the user sees the red border + error caption (good) but the preview keeps the
last valid styling with no "this block is being ignored" affordance beyond the caption. This is a
reasonable UX choice; flagging only because "what happens on parse failure" was an explicit review
question: **the answer is — error is shown, value is silently not applied.** No data is executed.
No fix required.

## Things I checked that are CLEAN (no finding)

- **Security of `parseSx`** (`parseSx.ts`): genuine non-executing recursive-descent parser. Rejects
  arrow/function values, calls, bare identifiers, template literals, IIFEs (tested at
  `parseSx.test.ts:45-64` with a spy proving no execution). Handles nested objects, arrays, trailing
  commas, quoted selectors, line/block comments, numbers (incl. exponent/negative), booleans, null,
  and the `({ … })` expression-wrapped form. Rejects non-object top-level values. Parse failure
  returns `{ value: undefined, error }` — never throws to the caller, never evals. 7/7 tests pass.
- **No `eval` / `new Function` / `innerHTML` / `dangerouslySetInnerHTML`** anywhere in the cluster.
- **Links**: every `href` in `chat-gallery/components.tsx` (27 entries) and the gallery section files
  resolves to an existing `docs/pages/x/react-chat/**.js`. No localhost, no branch names, no
  hardcoded preview URLs. `ChatOverviewRedirect.tsx` redirects to the stable `/x/react-chat/`.
- **Default exports**: every `*Playground.tsx`, `ComposerDemoShell.tsx` (named + default, per the
  "Fix composer demo shell default export" commit), `*Standalone*.tsx`,
  `ChatComponentShowcaseCard.tsx` has a default export.
- **Flat slot vocabulary**: registry `defaultRegistryMetadata` in `PlaygroundCard.tsx:126-365` and
  all playground `.tsx` use flat slot keys; no nested `slots={{ x: { … } }}`, `slots:null` /
  `features:false` used correctly (e.g. copilot demo `features={{ conversationHeader: false }}`,
  `slots={{ emptyState: CopilotEmptyState }}`). ChatBox registry slot keys
  (`conversationRoot`, `messageGroup`, `messageContent`, `composerRoot`, `composerInput`,
  `scrollToBottom`, `typingIndicator`, `emptyState`, …) all validated against
  `packages/x-chat/src/ChatBox/ChatBox.types.ts`. The registry is a curated _subset_ of the real
  flat slot set — intentional, not phantom keys.
- **Dark mode**: handled via `docsDarkModeSelector = ':where([data-mui-color-scheme="dark"], .mode-dark) &'`
  (`PlaygroundCard.tsx:124`), consistently applied to the controls panel, sx textarea, and chrome
  (the "Fix chat playground dark mode" approach). ChatBox itself inherits dark mode from the docs
  MUI theme provider.
- **Gallery thumbnails**: pure static SVG-schematic primitives (`chat-gallery/primitives.tsx`,
  `ChatBoxThumb.tsx` etc.) — they do **not** mount real `ChatBox`/`ChatProvider` instances, so the
  gallery page is cheap to render. Cards are a single `<Link>`, thumbnail wrapper is `aria-hidden`,
  accessible name comes from the `<Typography component="h3">`. A11y of the gallery is fine.
- **Imports**: playground providers/fixtures import only from public `@mui/x-chat` and
  `@mui/x-chat/headless` entry points — no `internals`/`/src/`/`/build/` deep imports.
- **`alpha()` import** in `ChatComponentShowcaseCard.tsx:2,67` is fine — it runs inside the docs
  brand theme (`applyDarkStyles` + `primaryDark` palette), not the x-chat CSS-vars alpha theme, so
  the known CSS-vars `alpha()` SSR pitfall does not apply here.
